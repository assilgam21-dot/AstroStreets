# AstroStreets — C++/DirectX 12 engine, Lua gameplay

A 2.5D pixelart game where **C++ owns the engine and the base classes** and
**Lua owns the per-entity behavior**. The engine opens a window, renders 64x64
pixelart sprites with DirectX 12 (including a 2D point-light model), runs a
camera, and steps a fixed-timestep simulation. Concrete entity logic — how a
specific thing animates and behaves — lives in Lua scripts you can edit without
recompiling.

## Threading model

The engine runs on **two threads** that communicate through small, lock-guarded
hand-off points — no shared mutable game state:

```
 main / render thread                 simulation thread
 ────────────────────                 ─────────────────
 owns D3D12 device + window           owns World + the Lua VM
 glfwPollEvents()                     fixed-timestep update (60 Hz)
   └─ feeds ──▶ Input (mutex) ──read─▶ World::Update(dt)
 draw latest frame  ◀── SnapshotBuffer (mutex) ◀── publish World::Render()
 present @ vsync
```

- **Why split this way:** the D3D12 device/swap chain and the GLFW window/event
  pump live on one thread, and Lua is single-threaded. So the render thread keeps
  the device + event pump, and the sim thread keeps the World and the Lua VM.
- **`Input`** (`Core/Input.h`) — the render thread writes key events; the sim
  thread reads held keys and drains discrete presses. One mutex.
- **`SnapshotBuffer`** (`Core/SnapshotBuffer.h`) — the sim thread publishes a
  finished `RenderFrame` (sprite draw list + HUD text + camera state); the render
  thread copies the latest. One mutex.

Rendering presents at vsync, simulation runs at a fixed 60 Hz step (catch-up
clamped to avoid a spiral of death).

## Rendering (DirectX 12)

`Engine/Renderer` is a small D3D12 backend: a flip-discard swap chain, one
command list synchronized with a per-frame fence, a root signature with two
pipeline states (sprites + HUD text), and persistently-mapped upload buffers for
the per-frame constants and dynamic vertices. HLSL is compiled at runtime with
`D3DCompile`; `Engine/D3DUtil.h` holds the manual texture-upload / barrier / SRV
helpers. Sprites are 64x64 PNGs in `assets/sprites/`, drawn pixelart-style and
point-sampled so they stay crisp at any zoom. The simulation refers to art by
`SpriteId`; `SpriteManager` maps each id to a D3D12 texture + SRV. The `Camera`
(follow / locked / animated focus, plus zoom) transform is applied in the vertex
shader.

### Lighting

The sprite pixel shader applies a forward 2D point-light model: an ambient term
plus a loop over the frame's lights (`RenderFrame::lights`), each attenuated by
`saturate(1 - d/radius)^2 * intensity`. Entities contribute lights through
`Entity::EmitLights`; the fire rune declares a warm, flickering `light` in
`scripts/entities/firerune.lua`, so the fire lights the floor around it.

Placeholder art is generated by `tools/gen_sprites.ps1` (a small logical grid
scaled up with no anti-aliasing). Drop in real PNGs with the same names to
replace it.

## Project layout

```
src/
  main.cpp                 wires the two threads together
  Core/                    pure data + threading primitives (no GL)
    Constants.h              TILE_SIZE
    Sprite.h                 a SpriteId + tint (a visual)
    SpriteId.h               stable handles for sprite assets
    Animation.h / .cpp       timed sequence of sprites (frame timers)
    Input.h / .cpp           thread-safe key state (string-named keys)
    RenderFrame.h            sprite/text/light draw lists + camera + ambient
    SnapshotBuffer.h         frame hand-off between sim and render threads
  Engine/                  windowing + DirectX 12 (render thread only)
    Window.h / .cpp          GLFW window (no GL context) + native HWND + events
    D3DUtil.h                manual D3D12 helpers (upload, barriers, SRVs)
    Renderer.h / .cpp        D3D12 lit-sprite pass + bitmap-font text pass
    SpriteManager.h / .cpp   loads PNGs into D3D12 textures, keyed by SpriteId
    Font.h / .cpp            builds the 8x8 bitmap-font atlas texture
    font8x8_basic.h          public-domain font data
  Game/                    gameplay (sim thread only)
    World.h / .cpp           owns entities + map + camera + queries
    GameLoop.h / .cpp        the fixed-timestep simulation loop
    Camera.h / .cpp          follow / locked / focus camera with zoom
    Scripting/
      ScriptEngine.h / .cpp  Lua VM (sol2): loads entity scripts, binds the API
    Entities/
      Entity.h / .cpp        base: id, position, Sprite + Animation, lifecycle
      Unit.h / .cpp          Entity + Health/Speed/AttackRange + combat hooks
      Player.h / .cpp        Unit: input-driven movement + melee
      Npc.h / .cpp           Unit: wander / chase / bite AI
      GameObject.h / .cpp    non-combat world object (e.g. chest)
      AreaTrigger.h / .cpp   rectangular trigger volume (enter/exit/stay)
      ScriptedAreaTrigger.*  AreaTrigger whose behavior is a Lua definition
      DamageInfo.h           a mutable damage packet (amount, school, attacker)
scripts/
  entities/firerune.lua    Lua-scripted fire rune (animation + behavior)
  spells.lua               reserved for the Lua spell layer (placeholder)
vcpkg.json                 dependency manifest
```

Dependencies (vcpkg manifest mode): `lua`, `sol2`, `glfw3`, `stb`. DirectX 12
(`d3d12`, `dxgi`, `d3dcompiler`) comes from the Windows SDK.

## Entity model and the C++/Lua split

The base classes live in C++ and own the common attributes and methods —
position, `Sprite` + `Animation`, the combat pipeline, and (over time) movement
primitives like chase / move-to / collision. The **behavior of a concrete
entity is a Lua script**, not a C++ subclass.

```
Entity                       id, (x,y), Sprite, Animation, lifecycle hooks
 ├─ Unit                     Health, Speed, AttackRange, AttackPower
 │   │                       combat pipeline + virtual hooks:
 │   │                         OnDamageTaken(Unit* attacker, DamageInfo* dmg)
 │   │                         OnDeath / OnHeal / OnKill
 │   ├─ Player               held-key movement, melee attack
 │   └─ Npc                  wander / chase / bite AI
 ├─ GameObject               non-combat object (OnUse)
 └─ AreaTrigger              rect volume, OnUnitEnter/Exit/Stay
     └─ ScriptedAreaTrigger  forwards hooks to a Lua entity definition
```

`ScriptedAreaTrigger` is the first reference implementation of the split: the
C++ side handles membership detection and exposes a bound API
(`unitsInside()`, `log()`, `Unit:takeDamage()`, `DamageSchool`, `Sprite`), and
`scripts/entities/firerune.lua` returns a table with the rune's frames and its
`onUpdate` / `onEnter` / `onExit` logic. The same pattern is the target for
projectiles (C++ move-to/collision, Lua decides the path and what happens on
impact) and other entities.

Combat flows `attacker.DealDamage(target, dmg)` → `target.TakeDamage(dmg)` →
`OnDamageTaken` (may absorb/modify) → apply → `Kill` → `OnDeath` / `OnKill`.

## Build & run

From Visual Studio: open `AstroStreets.slnx`, pick **x64 / Debug**, press F5.

From the command line:

```sh
msbuild AstroStreets.vcxproj /p:Configuration=Debug /p:Platform=x64
```

The first build downloads and compiles the vcpkg dependencies automatically.

## Controls

- **WASD / arrows** — move the player
- **Space** — melee attack the nearest unit in range
- **1 / 2 / 3 (or f)** — camera follow / locked / focus-on-area
- **z / x** — zoom in / out
- **Esc** — quit
